# 个人总结 — 王卓航（三元组构建与知识存储）

***

## 一、工作概述

在本项目中，我负责三元组构建与知识存储模块的设计与实现，完成了从结构化文物数据到知识图谱的构建工作。具体包括：MySQL 关系型数据库的表结构设计与数据存储、Neo4j 图数据库的知识建模与三元组抽取、以及跨数据库的数据一致性保障。最终构建了包含 **6,243 个文物节点**、**28 个朝代节点**、**3 个博物馆节点**及若干艺术家节点的知识图谱，实现了文物-博物馆-朝代-艺术家的多维关联查询。

***

## 二、主要工作内容

### 2.1 MySQL 关系型数据库设计

#### 表结构设计

设计了 6 张核心数据表，支撑文物数据的结构化存储：

| 表名               | 功能       | 核心字段                                                                                                             |
| ---------------- | -------- | ---------------------------------------------------------------------------------------------------------------- |
| museums          | 博物馆信息    | id, name, short\_name, country, city, website, collection\_url                                                   |
| dynasties        | 中国朝代表    | id, name\_zh, name\_en, start\_year, end\_year, description                                                      |
| artists          | 艺术家信息    | id, name\_zh, name\_en, birth\_year, death\_year, dynasty\_id, biography                                         |
| artifacts        | 文物主表     | object\_id, title\_zh, title\_en, time\_period, dynasty\_id, type, material, museum\_id, detail\_url, image\_url |
| artifact\_images | 文物图片     | id, artifact\_id, image\_url, image\_path, is\_primary, sort\_order                                              |
| artifact\_artist | 文物-艺术家关联 | artifact\_id, artist\_id, relationship\_type                                                                     |

#### 数据上传脚本

为三家博物馆分别编写了数据上传脚本，核心功能包括：

- **自动建表与数据导入**：支持 TRUNCATE + INSERT 和 APPEND 两种模式
- **连接断线重连**：检测 MySQL 错误码（2006/2013/2003），自动重建连接
- **批量提交**：每 50 条提交一次事务，平衡性能与可靠性
- **失败重试**：每条记录最多 3 次重试，记录失败行号
- **ID 分配策略**：

| 博物馆     | object\_id 范围 | museum\_id |
| ------- | ------------- | ---------- |
| 大都会博物馆  | 2501 \~ 5526  | 1          |
| 亚洲艺术博物馆 | 5527 \~ 6274  | 2          |
| 费城艺术博物馆 | 1 \~ 2500     | 3          |

> 三馆 object\_id 范围互不重叠，**零主键冲突**。

#### 艺术家提取与关联

- 从文物数据中自动提取艺术家信息，去重后分配唯一 ID
- 通过 `accession_number` 关联原始爬虫数据中的 Artist 字段
- 构建 `artifact_artist` 关联表，记录文物与艺术家的创作关系

### 2.2 Neo4j 知识图谱构建

#### 知识模型设计

设计了四类节点和三类关系，构成文物知识图谱的核心本体：

**节点类型**：

| 节点  | 标签       | 核心属性                                                         |
| --- | -------- | ------------------------------------------------------------ |
| 文物  | Artifact | object\_id, title\_zh, title\_en, material, type, image\_url |
| 博物馆 | Museum   | id, name                                                     |
| 朝代  | Dynasty  | id, name\_zh                                                 |
| 艺术家 | Artist   | id, name\_zh, name, birth\_year, death\_year                 |

**关系类型（三元组）**：

| 关系       | 类型            | 方向                 | 含义       |
| -------- | ------------- | ------------------ | -------- |
| 文物 → 博物馆 | COLLECTED\_BY | Artifact → Museum  | 文物被博物馆收藏 |
| 文物 → 朝代  | BELONGS\_TO   | Artifact → Dynasty | 文物属于某朝代  |
| 文物 → 艺术家 | CREATED\_BY   | Artifact → Artist  | 文物由艺术家创作 |

#### 数据导入方案

编写了完整的 Cypher 导入脚本（`Import_neo4j.md`），包含 6 个步骤：

1. **唯一约束创建**：为 Artifact、Museum、Dynasty、Artist 四类节点建立唯一性约束，防止重复导入并提升 MATCH 性能
2. **文物节点导入**：使用 `LOAD CSV` + `MERGE` 导入文物基本信息，支持 ON CREATE/ON MATCH 双模式更新
3. **收藏关系构建**：建立 Artifact → Museum 的 COLLECTED\_BY 关系
4. **年代关系构建**：建立 Artifact → Dynasty 的 BELONGS\_TO 关系
5. **作者关系构建**：基于 `artifact_artist_relation.csv` 建立 Artifact → Artist 的 CREATED\_BY 关系
6. **导入验证**：通过多跳查询验证文物-博物馆-朝代-艺术家的关联完整性

**性能优化**：

- 使用 `:auto` + `IN TRANSACTIONS OF 2000 ROWS` 分批提交，避免内存溢出
- 唯一约束加速 `MERGE` 操作，确保大规模数据导入的稳定性

#### 数据导出

为 Neo4j 导入准备了标准格式的 CSV 文件：

| 数据集       | 文件                                  | 行数    |
| --------- | ----------------------------------- | ----- |
| 亚洲艺术文物    | asian\_artifacts\_export.csv        | 748   |
| 亚洲艺术艺术家   | asian\_artists\_export.csv          | 若干    |
| 亚洲艺术关联    | asian\_artifact\_artist\_export.csv | 若干    |
| 大都会+费城文物  | artifacts\_export.csv               | 5,495 |
| 大都会+费城艺术家 | artists\_export.csv                 | 若干    |
| 大都会+费城关联  | artifact\_artist\_export.csv        | 若干    |

### 2.3 数据一致性保障

- **MySQL ↔ Neo4j 一致性**：验证清洗后数据与 Neo4j 导出数据 **100% 一致**，零数据丢失
- **朝代表跨馆一致性**：大都会与费城的 25 条朝代数据完全一致
- **museum\_id 全局一致**：三馆在清洗、翻译、数据库、Neo4j 各阶段的 museum\_id 完全一致
- **object\_id 零冲突**：三馆 ID 范围互不重叠

## 三、技术难点与解决方案

| 难点                | 解决方案                                                    |
| ----------------- | ------------------------------------------------------- |
| 三馆数据合并后主键冲突       | 设计分段 ID 分配方案（1\~2500/2501\~5495/5527\~6274），零冲突         |
| MySQL 大批量写入断线     | 检测错误码自动重连，每 50 条批量提交，失败记录 3 次重试                         |
| Neo4j 大规模数据导入内存溢出 | 使用 IN TRANSACTIONS 分批提交（2000 行/批）                       |
| 艺术家信息分散在各馆数据中     | 自动提取去重 + accession\_number 关联 + artist\_name\_to\_id 映射 |
| 关系型与图数据库数据一致性     | 统一数据源 + 导入后 COUNT 验证 + 106 项自动化测试                       |
| 朝代 ID 跨表一致性       | 统一朝代表（clean\_dynasties.csv），所有脚本引用同一映射                  |
